# Knowledge Distiller

The memory layer for Hermes-Agent. The goal is simple — **never solve the same problem twice, never look up the same environment twice**.

Conversations are full of reusable knowledge: resolved errors, verified environments, effective parameters. Without capture, that knowledge disappears when the session ends. This skill extracts it into a structured persistent store and retrieves it at the start of future sessions.

---

## ⚠ Operational Constraints — Read Before Acting

**The only permitted tool for all KB operations is `bash`** (to invoke the bundled Python scripts).

**Never use these tools on knowledge base files:**

| Prohibited tool | Why |
|---|---|
| `write_file` / `patch` / `edit` | Directly editing `kb.jsonl` or `index.md` will corrupt the store and may stall the stream mid-call |
| `execute_code` | All execution must go through the skill's bundled scripts (see path resolution below) |

**`index.md` is read-only from your perspective.** It is automatically regenerated by `update_knowledge.py` on every write. Any direct edit will be silently overwritten on the next update — and `write_file` calls targeting it have caused stream stalls in practice. Do not touch it.

Your role is: extract structured JSON from the conversation → pass it to the scripts via `bash`. All storage I/O, deduplication, and index rebuilding are handled deterministically by the scripts.

---

## Proactive Behavior Guide

Once loaded, act without waiting for an explicit user request — proactive operation is the core of Hermes-Agent's self-improvement loop.

**Retrieve — search before diagnosing:**
- Error text, stack trace, or non-zero CLI exit output is pasted into the conversation
- User says something like: 「前にも似たことやった気がする」「以前の設定を再現したい」「前にやったことある気がする」or in English: "we've hit this before", "what was the config last time"
- A specific model name, library, or version is mentioned in a technical question
- Starting a new task where prior art is plausible

**Store — save after resolution:**
- Problem is solved and working state confirmed
- New environment or model config verified
- Failure root cause identified — negative knowledge matters too
- User says: 「うまくいった」「解決した」「助かった」「記録しといて」or "that worked", "save this", "solved"

**Do not trigger** for casual chat, generic knowledge questions answerable from the web, or sessions dealing only with personal information.

## Core Design Philosophy

**Proximity over breadth.** For Hermes-Agent's typical workload (iterative trial-and-error), filtering by tag, category, and environment fingerprint outperforms broad vector search. The scripts combine lightweight TF-IDF with structured filters; `sentence-transformers` is used for semantic reranking when installed.

**LLM as distiller, scripts as storage.** Your job is *extraction* (conversation → structured schema) and *application* (reading results → forming a response). Storage I/O, deduplication, and indexing are deterministically delegated to the scripts. This reduces behavioral drift and improves reproducibility.

**Respect existing infrastructure.** If the user's environment has LLM-Wiki or another memory system, the scripts wrap it via the `cli` backend rather than creating a parallel store. See `references/schema.md` §6 for backend details.

## Workflow

### Resolving the Skill Directory

The scripts are bundled inside this skill's folder. Before running any command, establish `SKILL_DIR`:

```bash
# Default install path for Hermes-Agent skills — override via env var if installed elsewhere
SKILL_DIR="${KNOWLEDGE_DISTILLER_SKILL_DIR:-$HOME/.hermes/skills/knowledge-distiller}"
```

If the scripts are not found at `$SKILL_DIR/scripts/`, check that `KNOWLEDGE_DISTILLER_SKILL_DIR` points to the folder containing this `SKILL.md`.  
**Do not copy or recreate the scripts elsewhere** — use the originals in the skill directory.

---

### Phase A — Retrieve

Extract **keywords, category, and environment fingerprint** from the user's message and call the search script.

```bash
SKILL_DIR="${KNOWLEDGE_DISTILLER_SKILL_DIR:-$HOME/.hermes/skills/knowledge-distiller}"
python "$SKILL_DIR/scripts/search_knowledge.py" \
  --query "<3–7 invariant tokens from the error or topic>" \
  --category "error_resolution" \
  --tags cuda oom \
  --limit 5 \
  --format json
```

**Building `--query`:**
- Strip volatile parts: absolute paths, PIDs, memory addresses, timestamps
- Keep the invariant skeleton: exception class, library name, keyword combination
- Good: `"CUDA out of memory torch.distributed NCCL timeout"`
- Bad: `"/home/user/.cache/torch/lib/libc10.so line 42"`

**`--category` options:** `error_resolution` / `environment_setup` / `model_config` / `workflow` / `best_practice` — omit to search all.

**Reading results:** For each hit, compare `environment` fields against the current environment before applying `solution_steps`. A high match score does not guarantee reproducibility if the GPU, CUDA version, or framework version differs.

If no hits or all are low-relevance, retry with different vocabulary (up to 3 attempts), then treat as a new problem.

---

### Phase B — Solve

Proceed with the actual task, incorporating any knowledge retrieved in Phase A:
- State explicitly what past knowledge suggests and how the current situation differs
- If the user says "already tried that", treat their feedback as new negative knowledge for Phase C

---

### Phase C — Store

Call the update script when the task is complete, a root cause is identified, or the user asks to save.

```bash
SKILL_DIR="${KNOWLEDGE_DISTILLER_SKILL_DIR:-$HOME/.hermes/skills/knowledge-distiller}"
cat <<'JSON' | python "$SKILL_DIR/scripts/update_knowledge.py" --stdin
{
  "category": "error_resolution",
  "title": "PyTorch DDP NCCL timeout on bond0 interface",
  "summary": "On 2-node DDP (8× A100 each), NCCL timed out when NCCL_SOCKET_IFNAME was unset because bond0 caused MTU mismatch. Setting eth0 explicitly resolved it.",
  "environment": {"os": "Ubuntu 22.04", "cuda": "12.1", "node_count": 2, "gpu": "A100 80GB x8"},
  "error_symptoms": {
    "error_type": "RuntimeError",
    "error_message": "NCCL timeout after 1800s",
    "frequency": "always"
  },
  "solution_steps": [
    {"order": 1, "action": "export NCCL_SOCKET_IFNAME=eth0", "rationale": "bond0 causes MTU mismatch and latency"},
    {"order": 2, "action": "relaunch torchrun", "rationale": "apply the env var"}
  ],
  "outcome": "success",
  "confidence": 0.9,
  "tags": ["nccl", "ddp", "networking"]
}
JSON
```

**Record construction rules:**

- **`title`** — 8–60 chars; self-explanatory in a list view
- **`summary`** — 2–4 sentences covering "what happened, why, how fixed", self-contained
- **`environment`** — minimum fields needed to reproduce; omit irrelevant ones (don't fill empty strings)
- **`solution_steps`** — ordered list; `rationale` is required on every step — a step without reasoning is a memo, not knowledge
- **`confidence`** — user confirmed working = 0.9; observed but not independently verified = 0.6; hypothesis only = 0.3
- **`outcome: "failure"`** — save negative knowledge; "tried this, didn't work" cuts future iteration cycles

If the script warns of a duplicate (similarity ≥ 0.85), confirm with the user whether to merge (`--merge-with <id>`) or force-insert (`--force`).

**When required fields cannot be inferred from context:**

| Field | Cannot be determined | Action |
|---|---|---|
| `title` | Topic too vague | Ask the user for a 1-line summary before storing |
| `solution_steps` | Problem unsolved | Set `outcome: "failure"` and record what was attempted; omit the field rather than inventing steps |
| `environment` | No version info visible | Omit the field entirely — an empty object is worse than no object |
| `confidence` | Uncertain whether it worked | Default to `0.6`; note the uncertainty in `summary` |

**Never fabricate field values to satisfy schema.** A missing optional field is valid; an invented version number or fabricated step is corrupting noise.

## Storage — LLM-Wiki / Obsidian Vault (Default)

Knowledge records are stored as individual `.md` files (Obsidian-compatible) inside the LLM-Wiki vault, with a `kb.jsonl` index alongside for fast search.

**Directory resolution priority:**

| Priority | Env var | Resolved path |
|---|---|---|
| 1 | `HERMES_KB_DIR` | Use as-is |
| 2 | `WIKI_PATH` | `$WIKI_PATH/Hermes-KB/` |
| 3 | `WIKI` | `$WIKI/Hermes-KB/` |
| 4 | (fallback) | `~/.hermes/knowledge-base/` |

Since `WIKI_PATH` and `WIKI` are already set in Hermes-Agent's environment pointing to the Obsidian vault, **no additional configuration is needed** — the scripts pick them up automatically.

**Files written inside the KB directory:**

```
$WIKI_PATH/Hermes-KB/
├── kb.jsonl              ← JSONL index (fast structured search)
├── index.md              ← human-readable summary index (auto-regenerated)
└── records/
    ├── PyTorch_DDP_NCCL_timeout_a1b2.md   ← individual Obsidian notes
    ├── vLLM_0.4.2_Llama3_70B_e5f6.md
    └── ...
```

Each record in `records/` is a self-contained Obsidian note with YAML frontmatter (tags, category, confidence, outcome) and a human-readable Markdown body. They appear as regular notes in Obsidian and can be browsed, linked, and searched from within the vault.

To override the storage location explicitly:

```bash
export HERMES_KB_DIR="/custom/path/to/kb"
```

For external CLI backend (advanced), see `references/schema.md` §6.

**⚠ Obsidian edit / search sync:** The `.md` files in `records/` are Obsidian-readable, but **edits made directly in Obsidian do not propagate back to `kb.jsonl`**. Search results come from `kb.jsonl` only. To update a record, use `update_knowledge.py --merge-with <id>` — never edit the `.md` file directly.

## Error Recovery

| Symptom | Remedy |
|---|---|
| Too many hits (>20) | Add `--category` and `--tags`; narrow by `--since YYYY-MM-DD` |
| JSON validation error on store | Script returns the failing field name and expected type; check `references/schema.md` §5 |
| Backend unreachable | Scripts auto-fallback to `jsonl` and print `[warn] backend=X failed (...); falling back to jsonl` — data is saved locally, not to LLM-Wiki; pass `--no-fallback` to disable |
| JSONL slow (>100k records) | Use `--use-index` to activate tag inverted index at `$HERMES_KB_DIR/tags.idx.json` |
